La sezione critica
spiegata con un contatore

Due thread incrementano la stessa variabile, ognuno per 1000 volte. Il risultato atteso è 2000. Eppure non lo è quasi mai. Perché?

Il problema

Una classe banale: un contatore con un metodo increment(). Due thread chiamano increment() in parallelo, 1000 volte ciascuno. Alla fine ci aspettiamo 2000. In pratica leggiamo numeri come 1742, 1893, 1956. Quasi mai 2000.
public class Counter { private int count = 0; public void increment() { count++; // sembra atomico... ma non lo è } public int get() { return count; } }

L'istruzione count++ a livello di bytecode JVM è tre operazioni distinte:

// quello che la JVM fa davvero per "count++": 1. int tmp = count; // READ — leggi il valore corrente 2. tmp = tmp + 1; // COMPUTE — incrementa nel registro 3. count = tmp; // WRITE — scrivi il nuovo valore

Tra una sotto-operazione e l'altra lo scheduler del SO può sospendere il thread e dare la CPU all'altro. Ed è esattamente lì che nasce la race condition.

Suggerimento didattico: avviare prima senza synchronized con poche iterazioni (10–20) a velocità lenta per osservare l'interleaving e identificare il momento esatto del lost update. Poi rilanciare con synchronized attivo per vedere come scompaiono le anomalie.

La sezione critica

La sezione critica è il blocco di codice che non può essere eseguito da più thread contemporaneamente senza compromettere la consistenza dei dati condivisi. Nel nostro caso, la sequenza READ → COMPUTE → WRITE deve essere atomica: vista dall'esterno, tutta-o-niente.
✗ NON sicuro — race condition
public class Counter { private int count = 0; public void increment() { count++; } }
✓ Sicuro — mutua esclusione
public class Counter { private int count = 0; public synchronized void increment() { count++; } }

synchronized dice alla JVM: «prima di entrare qui, prendi il lock dell'oggetto. Solo un thread alla volta può eseguire questo blocco; gli altri si mettono in attesa.» In alternativa si può usare il blocco esplicito, utile quando la sezione critica è solo una parte del metodo:

public void increment() { synchronized (this) { // acquisizione del lock count++; // SEZIONE CRITICA } // rilascio del lock }

Da ricordare

01
Atomico ≠ una riga
Un'istruzione che sembra atomica nel sorgente (come count++) può essere scomposta in più operazioni a livello di bytecode/CPU.
02
Lo scheduler interrompe
Tra una sotto-operazione e l'altra il SO può sospendere il thread. Se la variabile è condivisa, qui nasce la race condition.
03
Sezione critica
È il tratto di codice che accede a dati condivisi e che deve essere eseguito da un solo thread alla volta per garantire correttezza.
04
Mutua esclusione
Il meccanismo (lock, monitor, synchronized) che impedisce a più thread di stare contemporaneamente nella stessa sezione critica.
05
Costo
La sincronizzazione ha un costo: serializza l'esecuzione e crea contesa. Si protegge solo ciò che serve, non più del necessario.
06
Non solo ++
Stessa logica per qualsiasi operazione composta: check-then-act, read-modify-write, lazy initialization. Pattern da proteggere.
